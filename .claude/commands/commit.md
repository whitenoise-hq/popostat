---
description: 변경사항을 Conventional Commits 규칙에 맞춰 커밋합니다
---

현재 작업 디렉토리의 변경사항을 검토하고 커밋한다.

## 절차

1. `git status`와 `git diff`(스테이징 전), 그리고 이미 스테이징된 변경이 있으면 `git diff --staged`로 변경 내용을 모두 파악한다.
2. 최근 커밋 로그(`git log --oneline -10`)를 확인해 이 프로젝트의 커밋 메시지 스타일·언어를 따른다.
3. 변경사항을 논리적 단위로 판단한다.
    - 서로 다른 목적의 변경이 섞여 있으면, 나눠서 커밋할지 사용자에게 제안한다.
    - 한 가지 논리적 변경이면 그대로 진행한다.
4. 관련 파일을 스테이징한다(`git add`). 의도하지 않은 파일이 포함되지 않도록 주의한다.
5. **Conventional Commits** 형식으로 메시지를 작성한다.
6. 커밋을 실행한다.
7. 커밋 결과(`git log --oneline -1`)를 보여준다.
8. 커밋이 완료되면 **"커밋이 완료되었습니다. 푸시할까요?"** 라고 사용자에게 묻는다. 사용자의 선택지는 Yes/No 로 선택지를 준다. 사용자가 동의하면 `git push`를 실행하고, 아니면 푸시하지
   않는다.

## 커밋 메시지 규칙

- 형식: `<type>: <subject>`
- type: `feat`(기능), `fix`(버그수정), `chore`(잡일·설정), `refactor`(리팩터링), `docs`(문서), `style`(포맷·스타일), `test`(테스트)
- subject: 간결하게, 무엇을 왜 했는지 드러나게. CLAUDE.md의 커밋 규칙을 따른다(한 커밋 = 한 논리적 변경).
- 프로젝트 기존 커밋이 한글이면 한글로, 영어면 영어로 일관되게.
- 필요하면 본문(body)에 상세 설명을 추가한다.

## 주의

- `git push`는 사용자에게 물어본 뒤 동의할 때만 실행한다(8번 절차). 묻지 않고 임의로 푸시하지 않는다.
- 사용자가 명시적으로 요청하지 않는 한 `git add .`로 전체를 무분별하게 담지 않는다. 변경 의도에 맞는 파일만 담는다.
- 비밀 키·`.env` 등 민감 파일이 스테이징 대상에 있으면 커밋하지 말고 경고한다.
- 커밋 메시지에 광고성 문구나 공동 작성자 서명을 임의로 추가하지 않는다.
